# Architecture Principles – Space Objects Data Platform

## Infrastructure Overview

All services run in Docker via a single `docker-compose.yml`. No external cloud dependencies.

### Services

### Infrastructure Rules

- **No `container_name:` directives** — containers use project-prefixed names (`space_object_platform-*`) to avoid conflicts with other projects on the same Docker host.
- **Port remapping** — Postgres: `15432:5432`, Kafka: `19092:9092` (avoids common port conflicts).
- **Single Docker network** — `space_net` bridge; containers resolve each other by service name.
- **Healthchecks** — Postgres and Kafka have healthchecks; dependent services use `condition: service_healthy`.

## Architecture Pattern: Medallion

### Bronze (Raw)
- As-ingested data; minimal transformation.
- JSONB columns preserve source API structure for fields with nested objects (e.g., `relative_velocity`, `miss_distance`, `estimated_diameter`, `parsed_record`).
- Uniqueness enforced by DB constraints (e.g., `ux_nasa_neo_event_raw_nk`).

### Silver (Normalized)
- Physical tables managed by **Apache Spark** jobs.
- Surrogate keys (`neo_sk`, `satellite_sk`) generated by PostgreSQL sequences.
- Idempotent: left anti-join prevents duplicate inserts.
- DQ checks run before every write; results stored in `meta.data_quality_run`.
- `meta.pipeline_run` row inserted (via psycopg2) at job start so DQ FK constraints are satisfied during execution.

### Gold (Analytics-Ready)
- **dbt-managed** tables in the `gold` schema.
- dbt silver models use `ephemeral` materialization — they become inline CTEs and do **not** create physical objects, avoiding naming conflicts with Spark-managed silver tables.
- `generate_schema_name` macro routes each model to its correct target schema (`gold`) regardless of the dbt profile's base schema.
- 51 dbt data tests cover not-null, unique, and accepted-value constraints.

## Data Warehouse Strategy

- **PostgreSQL as the single warehouse** — avoids multiple storage systems.
- All Spark JDBC writes use `stringtype=unspecified` to allow implicit varchar → uuid casting.
- Schema DDL is managed in `infra/postgres/init.sql` (CREATE TABLE IF NOT EXISTS pattern).
- Airflow, Spark, and dbt all target the same DB; each component manages its own schema(s).

---

## Orchestration

- **Airflow LocalExecutor** — single-node, appropriate for this Docker environment.
- 5 DAGs cover: NEO batch, SATCAT batch, bronze→silver Spark, dbt gold, streaming supervision.
- `streaming_supervision_dag` polls `meta.pipeline_run` every 10 minutes to detect stale or failed streaming producers.
- DAGs are volume-mounted (`airflow/dags/`) so changes take effect without image rebuilds.

---

## Observability

- Every Spark job writes a `meta.pipeline_run` row (psycopg2 at start, UPDATE at end).
- Every DQ check writes a `meta.data_quality_run` row with pass/fail counts and JSONB details.
- Structured Python logging (`%(asctime)s | %(levelname)s | %(name)s | %(message)s`) across all components.
- Prometheus scrapes Kafka and Postgres exporters; Grafana visualizes these.

---

## Synthetic Data

- `services/synthetic/generators.py` creates physically realistic NEO and SATCAT records using:
  - NEO: derived diameter from absolute magnitude (Bond albedo model), log-normal miss distance.
  - SATCAT: Kepler-derived orbital period, LEO/MEO/GEO/HEO orbit type distributions.
- Synthetic records flow through the same bronze → silver → gold pipeline as live data.
- Activated via `docker compose --profile synthetic up`.

---

## Security Considerations

- All credentials via environment variables (`.env` file, not committed to git).
- Postgres user (`space_user`) has schema-level privileges only.
- No internet-exposed ports by default; all host ports are for local development only.
- API keys are never logged.
